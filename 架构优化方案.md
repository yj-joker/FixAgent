# 架构优化方案

## 背景

当前架构为"线性意图识别 + 子 Agent ReAct"，相比旧的 LangChain LCEL 管道式架构，在灵活性和扩展成本上有明显优势，但在**结构化输出**和**可观测性**两方面有提升空间。

---

## 优化一：结构化输出 — 从"靠提示词祈祷"变成"API 强制 + Pydantic 校验"

### 问题本质

现在 MemoryAgent 在 prompt 里写了 60 行 JSON 格式说明，然后靠 `json.loads()` 解析。DashScope 的 `qwen-plus` 模型虽然听 prompt 话，但不是 100% 可靠——少一个逗号、多一行注释，JSON 就废了。

### 做法

**第 1 步：`LLMService` 支持 `response_format` 参数**

DashScope 兼容 OpenAI 的 `response_format: {"type": "json_object"}`，这会让模型在生成 token 时受语法约束，确保一定输出合法 JSON。目前 `chat()` 和 `chat_with_tools()` 都没传这个参数。

改动点只在 `llm_service.py`：

```python
# chat() 加一个参数
params = {
    ...
    "response_format": response_format  # {"type": "json_object"} 或不传
}

# chat_with_tools() 同理
```

**第 2 步：MemoryAgent 输出用 Pydantic 模型校验**

```python
from pydantic import BaseModel

class MemorySummary(BaseModel):
    new_facts: list[FactItem] = []
    superseded_ids: list[str] = []
    updated_preferences: list[PreferenceItem] = []
    updated_unresolved: list[UnresolvedItem] = []
    resolved_items: list[str] = []
    brief_summary: str = ""
```

`_extract_json()` 解析后用 Pydantic 校验，类型不对、字段缺失能立刻发现，而不是靠 fallback 返回一个空壳。这和旧架构的 `RetrievalChainOutput` / `DiagnosisChainOutput` 一脉相承，但不需要 LangChain 那套。

**第 3 步（可选）：管道上下文结构化**

现在 `_execute_full_pipeline()` 把上一个 Agent 的整段 `message` 文本塞给下一个 Agent 当 context：

```python
context["retrieval_result"] = retrieval_output.message  # 一大段自然语言
```

如果每个子 Agent 在 `metadata` 里放一份结构化的中间结果，管道传递的就是明确字段，而不是让下一个 LLM 重新从文本里提取。

## 优化二：ReAct 可观测性 — 从"黑盒"变成"步步留痕"

### 问题本质

现在 `chat_with_tools()` 的 ReAct 循环对调用方完全不透明。Agent 调了几个工具？每次都返回了什么？LLM 在哪一轮决定的"够了，不需要再查"？出问题时只能猜。

### 做法

**第 1 步：`chat_with_tools()` 返回 trace**

在 `llm_service.py:184` 的 for 循环里，每一步收集：

```python
trace = []

for iteration in range(max_iterations):
    step_start = time.time()
    response = await self._sync_chat_with_tools(...)
    
    tool_calls = response.get("tool_calls", [])
    if not tool_calls:
        trace.append({
            "iteration": iteration + 1,
            "action": "finish",
            "content_preview": response.get("content", "")[:100],
            "duration_ms": int((time.time() - step_start) * 1000)
        })
        return {**response, "trace": trace}   # 最终返回带上 trace
    
    observations = []
    for tc in tool_calls:
        result = await tool_handlers[func_name](**func_args)
        observations.append({
            "tool": func_name,
            "arguments": func_args,
            "result_summary": str(result)[:200]
        })
    
    trace.append({
        "iteration": iteration + 1,
        "action": "tool_call",
        "observations": observations,
        "duration_ms": ...
    })
```

返回值从 `{"content": "..."}` 变成 `{"content": "...", "trace": [...]}`——现有调用方不受影响，新增信息在 `trace` 字段。

**第 2 步：`BaseAgent.run_with_react()` 把 trace 写入 metadata**

```python
output.metadata["react_trace"] = response.get("trace", [])
output.metadata["react_iterations"] = len(response.get("trace", []))
```

这样任何 ReAct Agent 的执行过程都记录在 `AgentOutput.metadata` 里，前端可以通过 `/ai/chat` 的返回值拿到完整的推理链。

**第 3 步（可选）：流式推送 ReAct 步骤**

在 `/ai/chat/stream` 模式下，每完成一个 ReAct 步骤就推送一个事件：

```
data: {"type": "tool_call", "tool": "knowledge_retrieval", "args": {"query": "电动机过热"}}
data: {"type": "observation", "result": "找到3条相关知识"}
data: {"type": "tool_call", "tool": "graph_query_diagnosis_path", ...}
data: {"type": "token", "content": "根据分析..."}
```

这样用户不用干等——能看到 Agent 在"查资料→分析图谱→写回答"，而不是对着一个转圈。

---

## 改动量和风险

| 步骤 | 改动范围 | 风险 | 效果 |
|------|---------|------|------|
| `response_format` 参数 | `llm_service.py` 加一个参数 | 极低（向前兼容） | JSON 输出可靠性大幅提升 |
| MemoryAgent Pydantic 校验 | 新增一个小 Schema 类 | 低（fallback 仍在） | 类型安全，字段不缺失 |
| `chat_with_tools` 加 trace | `llm_service.py` 加 ~30 行 | 极低（只加字段） | 调试效率质的飞跃 |
| `run_with_react` 存 trace | `base_agent.py` 加 2 行 | 极低 | 全文 ReAct 链路可查 |
| 流式推送 ReAct 步骤 | `llm_service.py` + `api/main.py` | 中 | 用户体验明显改善 |

---

## 建议顺序

**先做前三项**（response_format + trace + metadata 存储），加起来不到 60 行改动，全是加字段不破坏现有逻辑，做完后调试和可靠性都有明显提升。

**流式推送放后面**，因为它要改 SSE 协议，需要和前端协调事件格式。
