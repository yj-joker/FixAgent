# FixAgent AI 模块架构

## 项目定位

**本项目是FixAgent系统的AI推理引擎，专注于故障诊断推理、知识检索和作业指引生成。**

> **重要变更**：基础后端功能（数据持久化、业务逻辑、会话管理、文件存储）由Java后端统一管理，本Python项目仅负责AI推理模块。

### 设计原则

| 原则 | 说明 |
|-----|------|
| 边界清晰 | Python只负责AI推理，不碰业务数据 |
| 松耦合 | AI模块通过HTTP接口与Java后端交互 |
| 国产化优先 | 使用国产组件（Redis、阿里云百炼等） |
| 性能优先 | 返回速度要快，支持异步和缓存 |
| 可追溯 | 推理过程可记录，支持审核 |

### Agent协作流程

```
用户输入
    │
    ▼
Java后端接收请求
    │
    ├── 查询会话历史（Redis ChatMemory）
    ├── 组装AIContext
    └── 调用Python AI服务
              │
              ▼
┌─────────────────────────────────────────┐
│      Orchestrator Agent（调度中枢）        │
│                                         │
│  意图识别 → 任务分解 → 分发执行           │
└─────────────────────────────────────────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
检索Agent  诊断Agent  作业Agent
    │         │         │
    └─────────┼─────────┘
              ▼
┌─────────────────────────────────────────┐
│           结果汇总输出                     │
│  ├── 自然语言（用户阅读）                 │
│  └── 结构化JSON（Java处理）              │
└─────────────────────────────────────────┘
              │
              ▼
Java后端处理响应
    ├── 存储对话历史
    └── 返回客户端
```

### 层级职责总结

| 层级 | 包 | 职责 | Spring 类比 |
|------|---|------|------------|
| **入口层** | api/ | 接收 HTTP 请求、参数校验 | @RestController |
| **调度层** | agents/ | 意图识别、分发任务、汇总结果 | @Service (业务编排) |
| **流程层** | chains/ | LCEL 链式调用定义执行流程 | Integration Flow |
| **工具层** | tools/ | 封装各种能力的调用接口 | @Component (工具类) |
| **服务层** | services/ | 封装外部服务（LLM、Redis、Neo4j）客户端 | Feign Client |
| **向量层** | embeddings/ | 文本/图像向量化 | Utility |
| **配置层** | config/ | 环境变量、配置管理 | @ConfigurationProperties |

### 包与包之间的关系

| 关系 | 描述 |
|------|------|
| api → agents | api 接收请求后调用 OrchestratorAgent 执行调度 |
| agents → chains | Agent 内部使用 Chains 定义的具体执行流程 |
| agents → tools | Agent 调用 Tools 执行具体任务（检索、查询、检测等） |
| agents → services | Agent 调用 Services 获取外部能力（LLM、向量、图谱） |
| chains → tools | Chains 编排 Tools 的调用顺序 |
| chains → services | Chains 调用 Services 实现具体功能 |
| tools → services | Tools 内部调用 Services 与外部交互 |
| tools → embeddings | Tools 内部调用 Embeddings 做向量化 |
| services → 外部 | Services 封装对百炼API、Redis、Neo4j 的调用 |
| schemas ← 所有 | 所有模块都依赖 Schemas 定义数据类型 |
| config → 所有 | 所有模块都从 Config 读取配置 |

## 数据流

### 完整故障诊断流程

```
1. Java后端接收用户请求
   POST /api/v1/chat
   {
     "message": "电动机轴承过热是怎么回事？",
     "session_id": "xxx"
   }

2. Java构建上下文
   ├── 从Redis获取ChatMemory
   ├── 组装AIContext
   └── 调用Python AI服务
              │
              ▼
3. Python AI模块执行推理
   OrchestratorAgent → RetrievalAgent → DiagnosisAgent → GuidanceAgent
   依次调用:
   ├── KnowledgeRetrievalTool (向量检索)
   ├── GraphQueryTool (图谱查询)
   └── LLMService (百炼API)
              │
              ▼
4. Python返回推理结果
   {
     "message": "根据分析，轴承过热可能原因...",
     "intention": "troubleshoot",
     "tools_used": ["knowledge_retrieval", "graph_query"],
     "reasoning": {
       "diagnosis": {...},
       "guidance": {...}
     }
   }

5. Java后端处理
   ├── 存储对话历史到Redis
   └── 返回响应给用户
```

### 请求/响应模型

**Request：极简设计，Java传递推理所需的最小数据**

```python
class AIChatRequest(BaseModel):
    session_id: str                      # Java传递，用于日志追踪
    message: str                          # 用户消息
    context: Optional[AIContext] = None   # Java提供的上下文
    images: Optional[List[str]] = None    # 图片URLs

class AIContext(BaseModel):
    """Java提供的推理上下文"""
    conversation_history: List[Message]    # 对话历史（Java管理）
    relevant_knowledge: List[str] = []    # 预检索的知识（可选）
    device_info: Optional[dict] = None     # 设备信息（可选）
```

**Response：返回结构化推理结果**

```python
class AIChatResponse(BaseModel):
    message: str                          # AI回复
    intention: Optional[str] = None       # 识别到的意图
    tools_used: List[str] = []             # 使用的工具
    reasoning: Optional[dict] = None       # 推理过程（可选）
```

## 技术栈

| 类别 | 技术选型 | 理由 |
|-----|---------|------|
| API框架 | FastAPI | 高性能、自动文档、类型安全 |

| LLM | 阿里云百炼 (Qwen-plus) | 国产、中文优化 |
| 向量数据库 | Redis Search | 高性能、国产友好 |
| 图数据库 | Neo4j | 图谱查询能力强 |
| Embedding | text-embedding-v4 + CLIP | 多模态 |
| 图片检测 | YOLOv8 | 速度快、预训练模型 |
| 图片分割 | SAM | 高精度分割 |
