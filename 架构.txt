# FixAgent AI 模块架构

## 项目定位

**本项目是FixAgent系统的AI推理引擎，专注于设备故障诊断推理、知识检索和维修作业指引生成。**

> 面向第十四届中国软件杯 A1 赛题："基于多模态大模型技术的设备检修知识检索与作业系统"。
> 目标平台：4核8GB 龙芯虚拟机（LoongArch + 麒麟OS），需轻量、高效。

基础后端功能（数据持久化、业务逻辑、会话管理、文件存储）由 Java 后端统一管理，本 Python 项目仅负责 AI 推理模块。

### 设计原则

| 原则 | 说明 |
|-----|------|
| 边界清晰 | Python 只负责 AI 推理，不碰业务数据 |
| 松耦合 | AI 模块通过 HTTP 接口与 Java 后端交互 |
| 国产化优先 | 使用国产组件（Redis、阿里云百炼等） |
| 性能优先 | 单 Agent 统一处理，避免多轮意图路由的额外延迟 |
| 可追溯 | 推理过程（ReAct trace）可记录，支持审核 |
| 轻量部署 | 不引入重型框架（如 LangGraph），原生 Python 实现 |

## 核心架构：FixAgent + ReviewAgent

### 架构选型

**ReAct（Reasoning + Acting）+ 3层确定性校验管线**

旧架构采用 Orchestrator + 意图路由 + 3 个子 Agent（检索/诊断/指引），存在以下问题：

| 问题 | 说明 |
|------|------|
| 意图路由延迟 | 每次请求多一轮 LLM 调用（约1-2秒），但实际子 Agent 的工具集高度重叠 |
| 子 Agent 割裂 | 检索/诊断/指引各自独立，无法在同一 ReAct 循环中灵活组合工具 |
| 路由准确率瓶颈 | 意图识别存在误分类风险，用户问题往往横跨多个"意图" |
| 代码冗余 | 3 个子 Agent 的系统提示词和工具集大量重复 |

新架构移除意图路由层，合并为单一 FixAgent：

- **FixAgent**：统一 ReAct Agent，持有全部工具（知识检索 + 图谱查询 + 设备搜索），LLM 在 ReAct 循环中自主决定调用哪些工具、调用顺序、调用次数
- **ReviewAgent**：3层确定性校验管线，零 LLM 调用，对 FixAgent 输出做校验（检索依据向量相似度 → 图谱路径 Neo4j 确认 → 安全规则引擎补全）
- **MemoryAgent**：保持不变，独立由 `/ai/memory/consolidate` 端点调用
- **RealtimeMemoryAgent**：实时记忆更新检测，每轮对话后异步调用，检测事实纠正和偏好变更（< 3s），不阻塞主对话流

### 为什么不用 LangGraph

| 考量 | 结论 |
|------|------|
| 功能需求 | FixAgent 是线性 ReAct 循环，无条件分支/并行/中断恢复需求 |
| 硬件约束 | 4核8GB 龙芯 VM，LangGraph + LangChain 依赖链庞大 |
| 现有基础 | BaseAgent 已完整实现 ReAct 循环 + SSE 流式 + function calling |
| 维护成本 | 原生实现更可控，无框架升级兼容风险 |

## 调用流程详解

### 故障诊断流程（FixAgent ReAct）

```
用户说: "电动机轴承过热是怎么回事？"

┌─────────────────────────────────────────────────────────────────────────┐
│  1. Java Backend 接收请求                                                 │
│     Controller → 组装请求参数 → POST /ai/chat                           │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  2. api/main.py 接收请求                                                  │
│     @Post("/ai/chat") → FixAgent.run_with_react()                       │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  3. FixAgent ReAct 循环                                                   │
│                                                                           │
│   ┌─ ReAct 循环 ──────────────────────────────────────────────────┐     │
│   │ Thought: "需要查图谱理解因果链路"                              │     │
│   │ Action: graph_query_diagnosis_path(keyword="电动机",            │     │
│   │         fault_name="轴承过热")                                  │     │
│   │         └──► GraphService → Neo4j                              │     │
│   │ Observation: 得到部件→故障→解决方案链路                        │     │
│   │                                                                │     │
│   │ Thought: "还需要检索相关技术细节"                              │     │
│   │ Action: knowledge_retrieval(query="电动机轴承过热原因")          │     │
│   │         └──► VectorService → Redis FT.SEARCH                   │     │
│   │ Observation: 检索到维修手册相关内容                             │     │
│   │                                                                │     │
│   │ Thought: "信息够了，输出诊断结论"                              │     │
│   │ Answer: 综合诊断结论（退出循环）                               │     │
│   └────────────────────────────────────────────────────────────────┘     │
│                                                                           │
│   所有 LLM 交互通过 chat_with_tools() → 阿里云百炼 API                    │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  4. ReviewAgent 3层确定性校验                                              │
│                                                                           │
│   输入：FixAgent 回答 + react_trace（工具调用记录）                      │
│   校验流程：                                                              │
│   ├── 第1层 检索依据校验：向量化回答中的事实性陈述，与检索结果比对     │
│   │   相似度 < 0.35 标记为"未验证"                                       │
│   ├── 第2层 图谱路径校验：故障-方案对应关系是否在 Neo4j 中真实存在      │
│   │   Cypher 查询确认，不可用时用 trace 结果兜底                        │
│   ├── 第3层 安全规则引擎：7类危险操作的关键词匹配，缺失则追加警告      │
│   └── 全部为确定性操作，零 LLM 调用                                      │
│                                                                           │
│   结果：标记未验证声明 / 自动追加安全警告 / 返回最终回答                │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  5. 返回 ChatResponse 给 Java Backend                                     │
│     { message, tools_used, react_trace, review_result, latency_ms }     │
└─────────────────────────────────────────────────────────────────────────┘
```

### 多模态处理流程

```
用户上传图片 + "这个部件是什么？怎么修？"
         │
         ▼
┌─────────────────────────────────┐
│  FixAgent 多模态预处理           │
│                                 │
│  Qwen-VL 图片理解：             │
│  → 识别设备/部件名称            │
│  → 描述可见故障现象             │
│  → 生成结构化文本描述           │
│                                 │
│  输出示例：                     │
│  "图中为摩托车发动机气缸体，    │
│   可见活塞顶部有明显烧蚀痕迹，  │
│   气缸壁有纵向拉伤痕迹"        │
└─────────────────────────────────┘
         │
         ▼
    图片描述 + 用户文本
    拼接为完整查询上下文
         │
         ▼
    进入 ReAct 循环
    （知识检索 + 图谱查询 + 诊断推理）

【为什么不用 YOLO/SAM】
- 赛题场景为"开放设备类型"（用户上传未知设备手册）
- YOLO 需要针对每种设备训练，无法覆盖所有设备类型
- Qwen-VL 作为多模态大模型，零样本即可理解未知设备图片
- SAM 图像分割对文本检索流水线无直接价值
```

### 记忆整理流程

```
Java 后端检测对话数量
   │
   ├── 会话对话数 < 30条 → 继续累积
   │
   └── 会话对话数 >= 30条 → 触发整理
              │
              ▼
        POST /ai/memory/consolidate
              │
              ▼
┌──────────────────────────────────────────────┐
│     MemoryAgent (function calling 模式)        │
│                                                │
│  1. 接收旧偏好 + 旧未完成事项 + 新对话        │
│  2. LLM 提取候选事实                           │
│  3. 调用 search_similar_facts 检索已有事实     │
│  4. 判断冲突（新增/更新/废弃）                 │
│  5. 输出结构化 JSON                            │
└──────────────────────────────────────────────┘
              │
              ▼
        返回 MemoryConsolidateResponse
        {
          new_facts, superseded_ids,
          updated_preferences, updated_unresolved,
          resolved_items, brief_summary
        }
              │
              ▼
        Java 存储：
        ├── new_facts → 事实库（向量表）
        ├── 偏好/未完成 → 摘要表
        └── superseded_ids → 标记旧事实无效
```

## 包与包之间的关系

| 关系 | 描述 |
|------|------|
| api → agents | api 接收请求后调用 FixAgent/ReviewAgent/MemoryAgent |
| agents → tools | FixAgent 在 ReAct 循环中调用 Tools（知识检索、图谱查询、设备搜索） |
| agents → services | Agent 调用 LLMService 进行 LLM 交互 |
| tools → services | Tools 内部调用 VectorService / GraphService 与外部交互 |
| tools → embeddings | KnowledgeRetrievalTool 内部调用 TextEmbedding 做查询向量化 |
| services → 外部 | Services 封装对百炼API、Redis、Neo4j 的调用 |
| schemas ← 所有 | 所有模块都依赖 Schemas 定义数据类型 |
| config → 所有 | 所有模块都从 Config 读取配置 |

---

## 模块职责

| 模块 | 职责 | 技术选型 |
|-----|------|---------|
| `api` | HTTP 接口、请求路由、参数校验 | FastAPI |
| `agents` | FixAgent 统一推理 + ReviewAgent 审核 + MemoryAgent 记忆整理 + RealtimeMemoryAgent 实时更新检测 | 原生 Python（ReAct + FC + 纯判断） |
| `config` | API Keys、模型配置 | python-dotenv |
| `embeddings` | 文本向量提取 | 百炼API text-embedding-v4 |
| `schemas` | 请求/响应数据模型 | Pydantic |
| `services` | LLM 调用封装、向量存储、图谱查询 | 百炼SDK / Redis / Neo4j |
| `tools` | 知识检索、图谱查询 | 多种 |

## 模块边界划分

### Java 后端职责

| 模块 | 职责 |
|-----|------|
| MySQL | 案例数据、设备数据、知识库业务数据 |
| Neo4j | 图谱业务数据（实体、关系）—— Java 端负责增删改 |
| Redis | 会话 ChatMemory、缓存、业务队列 |
| OSS | 文件存储（图片、文档） |
| Service层 | 全部业务逻辑（案例审核、设备管理等） |

### Python AI 模块职责

| 模块 | 职责 |
|-----|------|
| FixAgent | 统一推理：知识检索 + 故障诊断 + 维修指引 |
| ReviewAgent | 3层确定性校验管线（检索依据/图谱路径/安全规则） |
| MemoryAgent | 对话记忆整理压缩（定时整合） |
| RealtimeMemoryAgent | 实时记忆更新检测（事实纠正 + 偏好变更，每轮对话后异步调用） |
| Tools | 知识检索、图谱查询（只读） |
| Services | LLM 调用、向量检索、图谱查询 |
| Embeddings | 文本向量化 |

## API 接口

### AI 模块对外接口（Python 提供）

| 接口 | 方法 | 描述 | Java 调用场景 |
|-----|------|------|-------------|
| `/ai/chat` | POST | 对话接口（FixAgent → ReviewAgent） | 用户日常对话 |

## 技术栈

| 类别 | 技术选型 | 理由 |
|-----|---------|------|
| API 框架 | FastAPI | 高性能、自动文档、类型安全 |
| Agent 框架 | 原生 Python | 轻量，无重型依赖，适配龙芯 4核8GB |
| LLM | 阿里云百炼 (qwen-plus) | 国产、中文优化、远程 API 不消耗本地算力 |
| 多模态 | Qwen-VL (百炼 API) | 零样本图片理解，覆盖开放设备类型 |
| 向量数据库 | Redis Search | 高性能、国产友好 |
| 图数据库 | Neo4j | 图谱查询能力强 |
| Embedding | text-embedding-v4 | 百炼 API，中文语义优秀 |
