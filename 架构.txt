# FixAgent AI 模块架构

## 项目定位

**本项目是FixAgent系统的AI推理引擎，专注于故障诊断推理、知识检索和作业指引生成。**

> **重要变更**：基础后端功能（数据持久化、业务逻辑、会话管理、文件存储）由Java后端统一管理，本Python项目仅负责AI推理模块。

### 设计原则

| 原则 | 说明 |
|-----|------|
| 边界清晰 | Python只负责AI推理，不碰业务数据 |
| 松耦合 | AI模块通过HTTP接口与Java后端交互 |
| 国产化优先 | 使用国产组件（Redis、阿里云百炼等） |
| 性能优先 | 返回速度要快，支持异步和缓存 |
| 可追溯 | 推理过程可记录，支持审核 |

```

│   ├── __init__.py
│   └── main.py                   # API 路由定义、请求处理
│
├── agents/                       # Agent 实现（AI核心）
│   ├── __init__.py
│   ├── base_agent.py            # Agent 基类
│   ├── orchestrator_agent.py    # 调度 Agent（意图识别/任务分解）
│   ├── retrieval_agent.py       # 检索 Agent
│   ├── diagnosis_agent.py       # 诊断 Agent
│   ├── guidance_agent.py        # 作业 Agent
│   ├── memory_agent.py          # 记忆整理 Agent（对话摘要压缩）
│   └── intention/                # 意图识别模块（LLM+关键词兜底）
│       ├── __init__.py
│       ├── recognizer.py        # 意图识别器（LLM识别）
│       ├── prompts.py           # 识别提示词模板
│       └── fallback.py          # 关键词兜底策略
│
├── chains/                       # 路由映射（仅保留 orchestrator.py）
│   ├── __init__.py
│   └── orchestrator.py          # 意图→模式映射函数
│
├── config/                       # 配置管理（仅AI相关）
│   ├── __init__.py
│   └── settings.py              # 环境配置、API Keys
│
├── embeddings/                   # 向量化工具
│   ├── __init__.py
│   ├── text_embedding.py        # 文本向量化（百炼API）
│   ├── image_embedding.py        # 图像向量化（CLIP）
│   └── multimodal_embedding.py  # 多模态统一向量（CLIP）
│
├── schemas/                      # 数据模型（仅AI相关）
│   ├── __init__.py
│   ├── models.py                # 枚举和常量
│   ├── request.py               # 请求模型
│   └── response.py               # 响应模型
│
├── services/                     # 核心服务
│   ├── __init__.py
│   ├── llm_service.py           # 大模型服务封装（阿里云百炼）
│   ├── vector_service.py         # 向量数据库服务（Redis）
│   └── graph_service.py          # 图数据库服务（Neo4j）
│
├── tools/                        # 工具集
│   ├── __init__.py
│   ├── base_tool.py             # 工具基类定义
│   ├── knowledge_retrieval_tool.py  # 知识库检索工具
│   ├── graph_query_tool.py        # 图谱查询工具
│   ├── yolo_tool.py             # YOLO 目标检测（远程调用）
│   ├── sam_tool.py              # SAM 图像分割（远程调用）
│   └── document_tool.py          # 文档解析工具（远程调用）
│
├── .env                         # 环境变量
├── requirements.txt             # 依赖列表
└── 架构                         # 架构文档
```

### Agent协作流程

```
用户输入
    │
    ▼
Java后端接收请求
    │
    ├── 查询会话历史（Redis ChatMemory）
    ├── 组装AIContext
    └── 调用Python AI服务
              │
              ▼
┌─────────────────────────────────────────┐
│      Orchestrator Agent（调度中枢）        │
│                                         │
│  意图识别 → 任务分解 → 分发执行           │
└─────────────────────────────────────────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
检索Agent  诊断Agent  作业Agent
    │         │         │
    └─────────┼─────────┘
              ▼
┌─────────────────────────────────────────┐
│           结果汇总输出                     │
│  ├── 自然语言（用户阅读）                 │
│  └── 结构化JSON（Java处理）              │
└─────────────────────────────────────────┘
              │
              ▼
Java后端处理响应
    ├── 存储对话历史
    └── 返回客户端
```

### 包与包之间的关系

| 关系 | 描述 |

| schemas ← 所有 | 所有模块都依赖 Schemas 定义数据类型 |
| config → 所有 | 所有模块都从 Config 读取配置 |

### 请求/响应模型

**Request：极简设计，Java传递推理所需的最小数据**

```python
class AIChatRequest(BaseModel):
    session_id: str                      # Java传递，用于日志追踪
    message: str                          # 用户消息
    context: Optional[AIContext] = None   # Java提供的上下文
    images: Optional[List[str]] = None    # 图片URLs

class AIContext(BaseModel):
    """Java提供的推理上下文"""
    conversation_history: List[Message]    # 对话历史（Java管理）
    relevant_knowledge: List[str] = []    # 预检索的知识（可选）
    device_info: Optional[dict] = None     # 设备信息（可选）
```

**Response：返回结构化推理结果**

```python
class AIChatResponse(BaseModel):
    message: str                          # AI回复
    intention: Optional[str] = None       # 识别到的意图
    tools_used: List[str] = []             # 使用的工具
    reasoning: Optional[dict] = None       # 推理过程（可选）
```

**记忆整理专用模型**

```python
# Request：Java发送待整理的对话列表
class ConversationItem(BaseModel):
    seq: int                              # 对话序号
    role: str                             # 角色: user/assistant
    content: str                          # 对话内容

class MemoryConsolidateRequest(BaseModel):
    session_id: str                       # 会话ID
    conversations: List[ConversationItem] # 待整理的对话列表

# Response：返回结构化记忆摘要
class MemorySummary(BaseModel):
    core_question: str                    # 用户核心问题
    key_conclusions: List[str]            # 关键诊断结论
    user_preferences: List[str]           # 用户偏好/约束
    unresolved: List[str]                 # 未解决问题
    brief_summary: str                    # 200字内整体摘要

class MemoryConsolidateResponse(BaseModel):
    session_id: str                       # 会话ID
    summary: MemorySummary                # 整理后的记忆摘要
    original_count: int                   # 原始对话条数
    consolidated_at: str                  # 整理时间
```

## YOLO 模型选型与部署方案

### 选型结论：YOLOv8

| 维度 | 评价 |
|------|------|
| 成熟度 | Ultralytics官方维护，文档/社区/部署方案最丰富 |
| 微调 | 工业零件自定义训练最简单，几百张标注图即可 |
| 部署 | 支持导出 ONNX/TensorRT，ONNX Runtime纯CPU推理，LoongArch兼容 |
| 备选 | 如零标注数据起步，可先用YOLO-World（开放词汇，文字描述即可检测）跑通流程，积累标注数据后切回YOLOv8 |

### 为何不选其他版本

| 版本 | 不选理由 |
|------|---------|
| YOLOv5/v6/v7 | 已被v8全面超越 |
| YOLOv9 | PGI/GELAN架构部署复杂度大，社区支持少 |
| YOLOv10 | NMS-free推理更快，但生态不如v8成熟 |
| YOLOv11 | 最新但迁移有风险，v8成熟后可直接换权重升级 |

## YOLO 训练与推理分离

### 核心原则

训练和推理在不同环境执行，训练产物（ONNX模型文件）作为静态资源交付：

```
┌──────────────────────────────┐    ┌──────────────────────────────┐
│  本地GPU机器（Windows/Linux） │    │  龙芯虚拟机（LoongArch+麒麟） │
│                              │    │                              │
│  拍照片 → 标注 → 训练        │    │  FixAgent/                   │
│     ↓                        │    │    └── models/               │
│  yolo train → best.pt        │    │         └── best.onnx        │
│     ↓                        │    │                              │
│  yolo export → best.onnx     │    │  yolo_tool.py 中：           │
│     │                        │    │  onnx_session.run(image)     │
│     └──── 复制 ──────────────→    │  → [活塞, 气缸, 曲轴]        │
│                              │    │                              │
│  仅训练时使用，不部署         │    │  仅推理，不需要PyTorch       │
└──────────────────────────────┘    └──────────────────────────────┘
```

### 各阶段产物

| 阶段 | 产物 | 大小 | 说明 |
|------|------|------|------|
| 训练 | best.pt | ~100MB | PyTorch格式，含训练状态（梯度、优化器等） |
| 导出 | best.onnx | ~50MB | 纯推理格式，只含模型权重值，ONNX Runtime加载 |
| 推理 | 检测结果 | 几KB | JSON格式：[{name, bbox, confidence}, ...] |

### 为什么不在龙芯上训练

| 理由 | 说明 |
|------|------|
| 无GPU | 赛题提供4核CPU+8GB内存，纯CPU训练YOLO不可接受（几百张图可能训练数天） |
| LoongArch兼容性 | PyTorch对龙芯支持不完善，训练环境配置本身耗时巨大 |
| 训练是一次性的 | 标注完数据集→训练一次→导出，后续只需推理 |
| 环境隔离 | pyTorch依赖庞大（数GB），会污染AI推理服务的轻量环境 |

### yolo_tool.py 定位

- **不是训练脚本**，是ONNX推理的封装（继承BaseTool）
- 负责：加载best.onnx → 接收图片 → 推理 → 返回零件名列表
- ONNX Runtime仅几十MB，纯CPU运行，不依赖PyTorch

## YOLO 与 Qwen-VL 协作关系

### 两者定位

| | YOLO | Qwen-VL |
|------|------|---------|
| 模型类型 | 小模型（~3000万参数） | 大模型（~700亿参数） |
| 输入 | 图片 | 图片 + 文本 |
| 输出 | 零件名+坐标 | 场景语义描述 |
| 擅长 | 精确零件定位 | 整体状况理解 |
| 部署 | ONNX Runtime本地CPU | 阿里云百炼API（远程） |
| 作用 | 出精准零件标签 → 检索 | 出语义描述 → 诊断推理 |

### 协作流程

```
用户拍照 + "这里冒烟怎么回事"
         │
    ┌────┴────┐
    ▼         ▼
  YOLO     Qwen-VL
  输出：     输出：
  ["活塞",  "活塞附近有烧蚀痕迹，
   "气缸",   整体较老旧..."]
   "曲轴"]
    │         │
    └────┬────┘
         ▼
  拼成完整查询上下文：
  "用户问冒烟，涉及零件：活塞、气缸、曲轴。场景：活塞附近有烧蚀痕迹..."
         │
         ▼
  向量检索 + LLM生成诊断

分工原因：
  - YOLO出标签 → 向量检索更精准（标签 vs 自然语言描述，前者噪音小）
  - Qwen-VL出语义 → 诊断推理更全面（"烧蚀"这类故障现象不是YOLO能描述的）
```

## 数据集来源

### 赛题资源

赛题提供**摩托车发动机维修手册PDF**作为参考文档，其中包含：
- 发动机结构图/爆炸图（带零件编号和名称）
- 各部件维修步骤说明
- 故障现象与解决方案对照表

### 标注数据来源

| 来源 | 可行度 | 说明 |
|------|:---:|------|
| 维修手册结构图 | ★★★★★ | 从PDF提取发动机爆炸图，零件编号天然可作为标签 |
| 实拍摩托车发动机 | ★★★★★ | 找一台发动机，不同角度/光照拍300-500张 |
| 公开数据集 | ★★★ | Kaggle有发动机/摩托车零件检测数据集 |
| 数据增强 | ★★ 辅助 | 对已有图片做旋转、裁剪、调色，扩充3-5倍 |

### 训练规模

摩托车发动机零件种类约20-40种（活塞、气缸、曲轴、气门、火花塞...），几百张标注图即可启动微调。范围可控，不像通用场景需要万级数据。
