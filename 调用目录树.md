# FixAgent 调用目录树

## 整体架构

采用 **混合架构**：Orchestrator（编排器）负责意图识别与路由分发，子 Agent 内部使用 **ReAct 循环**（Thought → Action → Observation）自主决策工具调用和执行策略。

```
用户输入 → Orchestrator（意图识别）→ 路由分发
  ├── CHAT → 直接 LLM 对话
  ├── RETRIEVAL → RetrievalAgent (ReAct) → 检索→评估→追问→返回
  ├── DIAGNOSIS → DiagnosisAgent (ReAct) → 检索→追问→推理→诊断
  ├── GUIDANCE → GuidanceAgent (ReAct) → 检索→生成→校验→返回
  └── FULL → 串行: Retrieval → Diagnosis → Guidance → 汇总
```

## 架构关键变化

### 从编排器模式升级为混合ReAct架构

**之前（编排器模式）**：
```
用户 → 意图识别 → 分发子Agent → 固定流程 → 返回
```
问题：诊断是线性的一次性过程，无法追问用户、无法根据检索结果动态调整策略。

**现在（混合ReAct架构）**：
```
用户 → Orchestrator（意图分类）
  ├── 简单对话 → 直接LLM回复
  └── 复杂任务 → 子Agent ReAct循环
       └── Think → Act(调工具) → Observe(看结果) → Think → ... → 返回
```
优势：复杂诊断任务中LLM自主决定何时检索、何时追问、何时结束。

### ReAct 执行流程（以诊断为例）

```
用户: "发动机有异响"
  ↓
DiagnosisAgent.run_with_react()
  ↓
┌─ ReAct 循环 ─────────────────────────┐
│ Think: 信息不够，需要追问工况        │
│ Act: 向用户提问"什么工况？哪个部位？" │
│ Observe: 用户回复"加速时，气缸盖附近" │
│ Think: 可能是气门问题，检索知识库    │
│ Act: 调用 knowledge_retrieval 工具   │
│ Observe: 检索到3条气门相关文档       │
│ Think: 信息够了，给出诊断            │
│ Act: 生成诊断结果和维修建议          │
└──────────────────────────────────────┘
  ↓
返回诊断结果
```

## 数据流

### ReAct 诊断流程 ★已实现

```
用户输入
    │
    ▼
POST /ai/chat (mode=diagnosis)
    │
    ▼
OrchestratorAgent.run()
    │
    ├── _resolve_mode() → DIAGNOSIS
    ├── IntentionRecognizer.recognize() → TROUBLESHOOT
    └── _dispatch(DIAGNOSIS) → _execute_diagnosis()
        │
        └── DiagnosisAgent.run_with_react()
            │
            ├── _build_messages() → 系统提示词 + 用户输入
            ├── get_tools() → [knowledge_retrieval, ...]
            └── llm_service.chat_with_tools()
                │
                ┌── ReAct 循环 ──────────────────┐
                │ 1. LLM: "需要检索知识库"        │
                │ 2. tool_call: knowledge_retrieval│
                │ 3. 工具返回检索结果              │
                │ 4. LLM: "信息不够，追问用户"     │
                │ 5. LLM输出追问                   │
                │ 6. (用户回复后新一轮)            │
                │ ...直到LLM输出最终文本答案       │
                └─────────────────────────────────┘
                │
                ▼
            AgentOutput(message, tools_used, metadata)
```

### 对话整理流程（MemoryAgent）★已实现

```
Java 端检测到对话 ≥ 阈值
    │
    ▼
POST /ai/memory/consolidate
    │
    ▼
MemoryAgent.run()
    │
    ├── 构建消息（含旧偏好/未完成 + 新N条对话）
    ├── LLM function calling 循环 → 提取事实 + 冲突检测
    │   └── FactRetrievalTool._execute(queries)
    │       └── VectorService.search_by_text() (并发)
    ├── _extract_json() 解析LLM输出
    ├── _store_facts_to_vector() → 新事实向量化入库
    └── 返回 MemoryConsolidateResponse
```

## 异常处理架构

```
BaseTool.run()  ←───────── 模板方法，统一异常处理
    │
    ├── try: _execute() → ToolResult(success=True)
    ├── except ToolException: → ToolResult(success=False, error=ToolError)
    └── except Exception: → ToolResult(success=False, code="TOOL_ERROR")

BaseAgent.run()  ←───────── 模板方法，统一异常处理
    │
    ├── try: _build_messages() → _call_llm() → _process_response()
    └── except Exception: → AgentOutput(metadata={error_type, error_detail})

BaseAgent.run_with_react()  ←── ReAct模式，统一异常处理
    │
    ├── try: chat_with_tools() → AgentOutput
    ├── except RuntimeError: → "推理步骤超出限制" (max_iterations exceeded)
    └── except Exception: → AgentOutput(status="error")

OrchestratorAgent.run()  ←── 覆盖父类，统一异常处理
    │
    └── try: 意图识别 → 路由分发 → except → AgentOutput(status="error")
```
