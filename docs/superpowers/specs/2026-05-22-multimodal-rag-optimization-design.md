# 多模态 RAG 优化设计稿

## 1. 文档目标

本文档用于约束 FixAgent 当前多模态 RAG 链路的后续优化设计与实现边界。

本轮目标不是直接堆叠功能，而是先基于现有代码明确：

1. 当前已经具备哪些能力。
2. 当前哪些能力只实现了一部分。
3. 当前哪些能力仍然缺失。
4. 后续优化会影响哪些 Python、Java、Redis 和前端模块。
5. 每一类改动需要怎样验证。

本文档后续实现必须遵守一个基本原则：

> 向量只用于检索，不替代 PDF 原文件、文本原文、表格内容或图片原图。

---

## 2. 当前代码现状

### 2.1 当前主链路

当前 Python 侧 RAG 主链路已经形成基础闭环：

```text
用户导入 PDF
-> DocumentParserTool 解析 PDF
-> 拆出文本、表格、图片
-> KnowledgeService 负责向量化并写入 Redis
-> VectorService 负责 Redis 向量存储与向量检索
-> KnowledgeRetrievalTool 对外提供知识检索工具能力
-> FixAgent 调用知识检索工具组织回答
-> ReviewAgent 对回答做后置 grounding、图谱和安全校验
```

当前关键 Python 文件：

| 文件 | 当前职责 |
| --- | --- |
| `tools/document_tool.py` | PDF 解析、文本/表格/图片提取、章节归组 |
| `services/knowledge_service.py` | 知识入库编排、向量化、写入 Redis |
| `services/vector_service.py` | Redis 向量索引、存储、过滤和搜索 |
| `tools/knowledge_retrieval_tool.py` | 知识检索工具封装 |
| `agents/fix_agent.py` | 大模型工具调用和回答组织 |
| `agents/review_agent.py` | 回答后置校验 |

### 2.2 当前已实现能力

当前已确认具备以下能力：

1. PDF 文档可以被解析为文本、表格和图片结构。
2. 文本和表格可以被向量化后写入 Redis。
3. 项目已有 `ImageEmbedding` 能力。
4. 图片记录在存在 `image_url` 时可以走图片本体向量化。
5. 图片记录在缺少 `image_url` 时会退化为 caption 或默认说明文本向量化。
6. `category` 与 `tags` 已支持入库并支持检索过滤。
7. `FixAgent` 已能把知识检索作为工具调用。
8. `ReviewAgent` 已有回答后的 grounding 校验能力。

### 2.3 当前部分实现能力

以下能力已有基础，但链路仍未完整：

| 能力 | 当前已有 | 当前缺口 |
| --- | --- | --- |
| 图片本体向量化 | 已有 `ImageEmbedding`，有 `image_url` 时可用 | PDF 拆图到可访问 `image_url` 的正式链路未完整固化 |
| 图片元数据 | 已有页码、章节、caption、图片名、本地路径等字段 | 缺图片前后上下文、所属步骤、语义标题和语义总结 |
| 元数据过滤 | 已有 `category`、`tags` | 缺 `document_id`、`chunk_type`、设备类型、版本等更细过滤 |
| 低置信度处理 | 回答后已有 grounding 标记 | 检索阶段置信度、回答前证据不足控制仍缺 |
| 证据展示 | 检索接口能返回内容和 metadata | 聊天回答链路尚未形成完整结构化证据链输出 |

### 2.4 当前缺失能力

当前代码中未看到完整实现的关键能力：

1. 文本按维修手册章节、小节、步骤语义进一步切块。
2. 图片 `image_title` 与 `image_summary` 生成链路。
3. 图片 summary 文本向量与图片本体向量的双路召回。
4. 文本、关键词、表格、图片 summary、图片向量的多路召回框架。
5. 类型多样性控制和意图驱动排序。
6. reranker 精排链路。
7. 业务层统一的 `relevance_score` 语义。
8. 检索评测集和面向 RAG 的效果指标闭环。
9. 文档级去重、版本管理、索引重建和旧数据清理策略。

## 3. 设计目标

本项目的目标不是基础纯文本 RAG，而是面向维修手册的多模态 RAG。

目标系统应支持：

1. 用户用文本查维修步骤。
2. 用户用文本查参数表。
3. 用户用文本查相关图片。
4. 用户上传图片后辅助检索。
5. 文本、表格、图片结果都能追溯来源。
6. 证据不足时系统不输出确定性技术结论。

目标效果应优先满足：

1. 链路正确。
2. 证据可追溯。
3. 检索可评测。
4. 方案能在比赛环境和现有 Java/Python 分工下落地。

---

## 4. 目标数据流

### 4.1 入库链路

目标入库链路如下：

```text
用户上传 PDF
-> 文件存储保存 PDF 原文件
-> Python 解析 PDF
-> 拆出文本块、表格块、图片块
-> 文本块保留原文并做文本向量
-> 表格保留内容并做文本向量
-> 图片原图上传文件存储并得到 image_url
-> 图片本体做 ImageEmbedding
-> 多模态大模型生成 image_title 与 image_summary
-> image_summary 做文本向量
-> Redis 保存向量、原内容引用、URL 和元数据
```

### 4.2 查询链路

目标查询链路如下：

```text
用户提问
-> 大模型判断是否调用检索工具
-> 检索系统识别查询意图
-> 多路召回候选
-> 统一分数语义
-> 合并候选
-> 控制类型多样性
-> reranker 精排
-> 计算检索置信度
-> 返回文本、表格、图片证据
-> 大模型基于证据回答
-> 证据不足时降级、追问或明确提示依据不足
```

## 5. 优化项设计

### 5.1 原数据保留

必须保留：

1. PDF 原文件。
2. 文本原文。
3. 表格内容。
4. 图片原图。

向量只承担相似检索职责，不承担内容展示和内容恢复职责。

### 5.2 图片存储地址统一

正式链路中，PDF 拆出的每张图片都应拿到前端可访问的 `image_url`。

文件存储实现不能写死为唯一方案。设计上应允许：

1. MinIO 作为主存储实现。
2. 本地文件存储作为比赛环境或部署受限时的兜底实现。

只要上层能得到稳定可访问的 `image_url`，RAG 主链路不应依赖底层存储实现细节。

### 5.3 多模态拆分入库

文本、表格、图片必须显式区分内容类型。

每条知识记录至少应保留：

1. 来源文档信息。
2. 页码信息。
3. 章节信息。
4. `chunk_type`。
5. `category`。
6. `tags`。

后续可扩展：

1. `document_id`。
2. `device_type`。
3. `manual_type`。
4. `document_version`。

### 5.4 文本切块策略增强

当前文本主要按页入库。

后续增强目标不是简单改成固定字数切块，而是尽量让 chunk 对齐维修知识单元，例如：

1. 章节。
2. 小节。
3. 操作步骤。
4. 参数说明。
5. 注意事项。

切块设计需要避免：

1. 一个维修步骤被无意义拆散。
2. 一个 chunk 混入多个无关主题。
3. 页边界直接打断查询所需上下文。

### 5.5 图片上下文增强

图片记录除基础字段外，应补充用于理解图片语义的上下文。

优先考虑：

1. 图片前后文本。
2. 所属步骤。
3. 相邻说明文字。
4. 所在章节和页码。

这些上下文既可服务图片 summary 生成，也可服务检索排序和证据展示。

### 5.6 图片本体向量化链路补全

当前项目已存在 `ImageEmbedding`，因此后续任务是补全 PDF 图片正式入库链路。

目标状态：

1. PDF 拆出的图片拥有 `image_url`。
2. 图片本体通过 `ImageEmbedding` 入库。
3. 图片记录保留前端展示所需 URL。
4. caption 文本向量化仅作为兼容性降级路径，不作为正式主链路。

### 5.7 图片语义向量化

多模态大模型应为图片生成：

1. `image_title`。
2. `image_summary`。

其中：

1. `image_title` 用于短标题展示和快速识别。
2. `image_summary` 用于语言语义检索和 RAG 上下文理解。

`image_summary` 应再走文本向量化链路。

### 5.8 图片双路召回

文字查图片时至少要考虑两条通道：

1. 查询文本向量对比图片本体向量。
2. 查询文本向量对比图片 summary 文本向量。

两条通道共同命中同一图片时，可以作为置信度特征之一，但不能直接替代 reranker 和结果校验。

### 5.9 多路召回

后续检索候选来源应支持逐步扩展：

1. 文本向量召回。
2. 关键词召回。
3. 表格召回。
4. 图片 summary 召回。
5. 图片向量召回。

召回阶段目标是扩大候选覆盖，最终排序不应直接等同于第一阶段召回顺序。

### 5.10 类型多样性控制

泛查询不能默认被单一数据类型占满。

系统需要支持：

1. 文本、图片、表格分别获得召回机会。
2. 仅在结果相关时返回对应类型。
3. 不以机械凑齐类型为目标。

### 5.11 意图驱动排序

排序策略要识别用户意图：

| 查询意图 | 优先类型 |
| --- | --- |
| 步骤、故障原因、怎么拆、怎么装 | 文本 |
| 图片、示意图、位置图、结构图 | 图片 |
| 参数、型号、规格、扭矩、数量 | 表格 |
| 泛词查询 | 混合结果 |

排序不能固定为某一种类型永远在前。

### 5.12 Reranker 精排

后续排序采用两阶段设计：

1. 召回阶段找全候选。
2. reranker 阶段排准候选。

reranker 接入后，最终展示顺序应以精排结果为主，而不是原始向量距离顺序。

### 5.13 分数语义统一

系统业务层统一使用：

```text
relevance_score 越大，表示越相关
```

同时保留调试字段：

1. `raw_score`。
2. `raw_score_type`。
3. `relevance_score`。

多路召回场景下，即使分数方向统一，仍需根据通道设计融合、归一化或重排策略。

### 5.14 检索置信度判断

检索系统应提供结构化置信度依据，而不是把判断完全交给大模型。

置信度可综合：

1. 召回分数。
2. reranker 分数。
3. 命中类型是否符合查询意图。
4. 图片双路是否共同命中。
5. 是否存在多条一致证据。
6. 是否满足查询中的精确条件。

后续检索结果可考虑补充：

1. `matched_types`。
2. `retrieval_confidence`。
3. `confidence_reason`。
4. `candidate_count`。
5. `rerank_score`。

### 5.15 低置信度兜底

后续需要在回答前补充证据不足控制：

1. 检索弱时降级返回证据。
2. 信息不足时追问用户。
3. 证据不足时明确提示。
4. 不让大模型在无依据时输出确定性技术结论。

这项能力是对现有 `ReviewAgent` 后置 grounding 校验的前置补全，不是替代它。

### 5.16 元数据过滤增强

在现有 `category/tags` 基础上，后续补充更细粒度过滤能力。

优先级较高的过滤维度：

1. `document_id`。
2. `chunk_type`。
3. `device_type`。
4. `document_version`。

### 5.17 检索评测集闭环

必须建立面向本项目的 RAG 评测集。

评测问题至少覆盖：

1. 步骤类问题。
2. 参数类问题。
3. 图片类问题。
4. 故障原因类问题。
5. 泛查询。
6. 图文混合查询。

评测结果至少跟踪：

1. Recall。
2. TopK 命中率。
3. reranker 前后排序变化。
4. 最终回答正确率。

### 5.18 去重与版本管理

后续知识入库必须支持工程生命周期治理。

需要设计：

1. 文档 ID。
2. 文件哈希。
3. 文档版本。
4. 入库批次。
5. 删除某文档全部 chunk 的能力。
6. 重建索引和清理旧索引的策略。

### 5.19 证据链展示

最终回答链路应支持返回和展示证据：

1. 命中文本片段。
2. 命中表格。
3. 命中图片及 `image_url`。
4. 来源 PDF。
5. 页码。
6. 章节。

### 5.20 大模型职责收口

大模型负责：

1. 理解问题。
2. 判断是否检索。
3. 必要时改写查询。
4. 基于证据组织回答。
5. 在证据不足时表达不确定性。

检索系统负责：

1. 过滤。
2. 召回。
3. 融合。
4. 排序。
5. 分数统一。
6. 置信度依据。

## 6. 模块影响面

### 6.1 Python 侧影响

| 模块 | 预期影响 |
| --- | --- |
| `tools/document_tool.py` | 文本切块增强、图片上下文提取、后续更稳定的 chunk metadata |
| `services/knowledge_service.py` | 图片 URL 主链路、图片 summary 入库、双索引或双记录组织 |
| `services/vector_service.py` | 统一分数语义、扩展过滤字段、可能支持多通道搜索 |
| `tools/knowledge_retrieval_tool.py` | 多路召回、类型策略、意图排序、置信度返回 |
| `schemas/*` | 检索结果、证据、分数字段和新增元数据结构 |
| `agents/fix_agent.py` | 使用结构化证据和低置信度结果回答 |
| `agents/review_agent.py` | 与检索置信度协作，保留后置 grounding 校验 |

### 6.2 Java / 文件存储影响

Java 或独立文件存储层需要明确：

1. PDF 原文件如何保存并返回地址。
2. Python 拆出的图片由谁上传存储。
3. MinIO 与本地存储是否共用统一存储接口。
4. Python 侧拿到的是公网 URL、内网 URL，还是后端代理 URL。

### 6.3 Redis 影响

以下改动可能影响 Redis 数据结构或索引：

1. 新增过滤字段。
2. 新增分数语义字段。
3. 图片 summary 检索记录组织方式。
4. `document_id` 和版本字段。
5. 多类型检索时是否需要区分索引或统一索引字段。

### 6.4 前端影响

前端后续要支持：

1. 图片结果展示。
2. 表格结果展示。
3. 证据来源展示。
4. 低置信度提示。
5. 不同结果类型的视觉区分。

## 7. 数据重建与兼容性

以下优化大概率需要重建已有向量数据：

1. 图片从 caption 降级向量切换到图片本体主链路。
2. 为图片新增 `image_title` 与 `image_summary`。
3. 文本切块策略从按页升级到步骤/小节级别。
4. 新增文档 ID、版本、chunk 级过滤字段。

需要保留兼容性策略的地方：

1. 老数据是否允许继续检索。
2. 新旧分数字段如何共存。
3. 图片没有 URL 时是否继续允许降级。
4. 存量文档如何迁移或重导入。

## 8. 分阶段实施建议

### 阶段一：先保证链路正确

1. 明确 PDF 图片上传后得到 `image_url` 的协作方式。
2. 确保 PDF 图片正式入库优先走图片本体向量化。
3. 补清楚图片结果返回结构和前端展示依赖。
4. 固化图片 summary 所需输入上下文和数据字段。

### 阶段二：提升检索效果

1. 实现图片 `image_title` 与 `image_summary`。
2. 实现图片双路召回。
3. 实现多路召回基础框架。
4. 引入类型策略和意图排序。
5. 接入 reranker。
6. 统一分数语义。

### 阶段三：补可靠性和工程治理

1. 检索置信度。
2. 低置信度兜底。
3. 检索评测集。
4. 细粒度元数据过滤。
5. 去重与版本管理。
6. 证据链展示。

## 9. 测试与验证要求

### 9.1 模块测试

每次修改必须覆盖与风险相匹配的测试：

1. 文档解析测试。
2. 图片 URL 入库测试。
3. 图片本体向量化路径测试。
4. 图片 summary 入库测试。
5. 分数字段转换测试。

7. 多路召回合并和类型策略测试。
8. 低置信度路径测试。

### 9.2 集成验证

后续必须验证：

1. Java 文件存储到 Python 入库是否完整连通。
2. Python 入库到 Redis 是否完整连通。
3. 检索接口是否能返回文本、表格和图片。
4. 聊天回答是否能消费结构化证据。

### 9.3 效果评测

不能仅凭单个查询样例判断检索变好。

至少要建立一批固定问题，并对比：

1. 原始 TopK 检索效果。
2. 图片 summary 增强后的效果。
3. 多路召回后的效果。
4. reranker 后的效果。

## 11. 禁止事项

1. 禁止把当前按页文本块误说成固定字数切块。
2. 禁止把已有 `ImageEmbedding` 误说成图片向量化能力不存在。
3. 禁止重复实现现有 `category/tags` 过滤。
4. 禁止把现有 `ReviewAgent` grounding 校验误说成完全没有低置信度相关能力。
5. 禁止用向量替代原文、表格内容或图片原图。
6. 禁止删除原始调试分数只保留转换分数。
7. 禁止没有评测就声称准确率显著提升。
8. 禁止在没有迁移方案时大范围改 Redis 结构。
9. 禁止把 MinIO 写死为唯一存储依赖。
10. 禁止证据不足时让大模型输出确定性技术结论。
